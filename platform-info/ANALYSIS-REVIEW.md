## 7. 此前分析中「说对了」的部分

1. **图例方向**：`-` = 漏输出，`+` = 多输出（后期已纠正早期误读）。
2. **order-by**：NULL 排序逻辑错误 → `d364437` 修复 → 平台 +20。
3. **group-by / expression / view / ctas / unique / text / multi-index**：大方向与 768c08c 仍一致。
4. **修复策略**：小步 commit、记录 mapping、本地 socket 协议复现——流程正确，缺的是 **与最新 detail 对齐**。
5. **用户纠正总分**：强制按所得分相加，纠正了 250 误写——此后基线改为 160/240 才自洽。

## 8. 此前分析中「说错了」的部分

1. **总分 250**：无 detail 支撑，与 10 题满分（160 分）矛盾。
2. **768c08c 相对退步**：在错误基线上得出 -10，实际 **+80**。
3. **function 0 分 = 平台没编出来**：同次提测 order-by/big-query 等已满分，说明二进制是新构建；function 更可能是代码路径问题。
4. **5 个 commit 可 +80~+100**：实际仅 order-by +20 可明确归因；null/alias/update-select/function 均为 0。
5. **alias 修 t2.* 即可过 alias 题**：平台换题后该修复与得分无关（至少在当前 detail 下）。
6. **null 修 IS NULL 即可过 null 题**：平台换考 MIN 聚合 NULL。
7. **big-query/big-write 仍需修**：在 768c08c 已满分，继续投入是浪费（除非回归）。

## 9. 后续分析检查清单

在写下一版修复计划或复盘前，逐项核对：

- [ ] 总分 = 25 题「所得分」之和，不用平台页眉或其他口述分数
- [ ] 失败分析引用的是 **当前基线 detail** 里的 SQL，不是历史 commit 的 detail
- [ ] 同题名变更时，单独建「SQL 对比」小节（如本文第 4 节）
- [ ] 本地 `_local/*.sql` 断言与当前 detail 一致
- [ ] commit 归因：只有「所得分 Δ ≠ 0」且能对应代码改动时才写 ✅
- [ ] 不得分 ≠ 修复无效：先检查是否换题、是否只覆盖子用例
- [ ] 不将「本地通过」写成「平台预期通过」
